### systemEvent Root Cause Analysis — M3 Source Code Audit (2026-06-09 18:47 HKT)
Josh asked M3 sub-agent to deep-dive whether skill_reviewer can support systemEvent mode. M3 read OpenClaw runtime source and found structural incompatibility:

**Mechanism 1 — Deduplication locks the queue:**
```js
// system-events-BMZOXQOE.js line 53-67
enqueueSystemEvent(text, {
    contextKey: `cron:56e09616-...`  // fixed per job ID
})
```
Only the 1st run enqueues; subsequent runs with same contextKey are silently dropped. This explains why only 4 out of ~12 expected runs appeared, and all were the first of each backoff window.

**Mechanism 2 — systemEvent is a hint, not a command:**
```js
// session-system-events-COgs63Nj.js line 49-67
// Injected as context into LLM prompt only
// LLM decides whether to act on it
```
Main session LLM may treat it as informational and skip it. No forced execution path.

**Mechanism 3 — Main session liveness:**
systemEvent runs need main session to be active. High-frequency crons run during off-peak hours (13:00-15:00 HKT) when LLM may be idle or processing other events.

**Contrast: 9 systemEvent cron jobs that DO work:**
All daily single-run jobs scheduled during active user hours (01:00-08:00 HKT):

→ These work because main session is processing them during active windows

**Contrast: 6 systemEvent cron jobs that FAILED:**
All high-frequency or daytime:

- Anomaly Monitor (06:30/18:30) — 7ms
