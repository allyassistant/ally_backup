## Pipeline Recommendations

### What's working

1. **Real signals → real skills.** The reviewer is detecting genuine patterns (cron model fallback debugging, daily synthesis workflow, X-link Obsidian archival) and producing useful workflow descriptions. This is not noise.

2. **Frontmatter discipline is solid.** All 21 skills have `name`, `description`, `status`, `source`, `provenance`, `generatedAt`. Consistent metadata enables the curation pipeline to work.

3. **5 high-quality skills are ready to use as-is:** `cron-model-selection-verification`, `issue-conclusion-overturn-cleanup`, `multi-session-resumption`, `parallel-subagent-implementation`, `skill-curation-pattern`, and arguably `openclaw-config-schema-debugging`. These would actually help a future session.

4. **The "active" promotion gate exists.** Some skills have been promoted to `active` and symlinked to `skills/_learned_*/`. The mechanism is working — but the quality bar for promotion is broken (see below).

### What's not working

1. **🚨 Write output is being truncated mid-workflow.** 4 of 21 skills (19%) are truncated stubs:
   - `cron-job-testing` — 32 words
   - `cron-thin-executor-migration` — 133 words, cut off at "Script 結構："
   - `llm-call-execfile-migration` — 36 words, cut off at "常見的錯誤模式是："
   - `route-enforcer-plugin-debugging` — 70 words, cut off at "繪製 execution flow"
   
   These are not "weak content" — they are **incomplete writes**. The skill_reviewer.js pipeline is calling `write` with content that gets cut off, and there's no post-write integrity check.

2. **🚨 Stubs are being promoted to `active` and symlinked.** `cron-thin-executor-migration` (133 words) and `llm-call-execfile-migration` (36 words) are in `skills/_learned_*/` with `active` status. The pipeline is shipping broken skills into `<available_skills>` injection.

3. **No minimum body length enforcement.** The `skill-curation-pattern` describes a `bodyLength < 200 → draft` rule for junk detection, but the existing rule does NOT prevent stubs from being created. It only catches already-existing stubs. The pipeline should **reject writes that result in < 200 words of body content** before they hit disk.

4. **No "Workflow section" completeness check.** Skills with workflow steps that don't end on a complete step (e.g., "Script 結構：" with no actual structure) are being accepted. The pipeline should verify the body has a complete "## Workflow" section with N numbered steps that all end with a period or punctuation.

5. **Topic overlap is creating parallel skills.** `cron-job-testing` + `cron-failure-investigation` + `cron-model-selection-verification` + `cron-thin-executor-migration` are 4 cron-related skills, of which 2 are stubs and 1 is hyper-specific. The reviewer should detect when 2+ skills in the same domain are being generated in the same week and consolidate.

6. **Cross-references are weak.** Only 2 of 21 skills have an explicit "Related Skills" section. The `skill-curation-pattern` itself documents this as a curation step but the generator doesn't follow it.

### Suggested improvements

1. **Add post-write integrity gate in `skill_reviewer.js` (or its bot counterpart).** Before writing:
   - Reject if body word count < 200
   - Reject if "## Workflow" section has < 3 steps
   - Reject if last workflow step doesn't end with punctuation
   - Reject if the last code block is unclosed
   
   Implementation: a post-write `node --check` style script (e.g., `node scripts/validate_skill_file.js <path>`) that runs as part of the cron job.

2. **Demote all currently-stuck stubs.** Run a one-time `find skills-learned/ -name SKILL.md | xargs wc -w | awk '$1 < 200'` and either delete or mark as `status: archived` with note "stub, needs regeneration".

3. **Tighten the `active` promotion gate.** The current logic (per `skill-curation-pattern` step 9) is `bodyLength >= 200 AND status == 'active' → promote with symlink`. This is too lenient. Add: must have Pitfalls section; must have at least 5 numbered workflow steps; must end with a non-truncated last step.

4. **De-dup overlapping skills at the review stage.** Before creating a new skill, check for existing skills in the same domain (cron, sub-agent, model-related, knowledge-curation, etc.). If overlap > 60%, refuse to create a new one — either update the existing or merge.

5. **Add a "domain-specific" status flag.** Rapaport-email-summary is intentionally narrow. Add `status: domain-specific` (instead of `active` / `draft`) so the curator knows not to flag it for low generalizability.
