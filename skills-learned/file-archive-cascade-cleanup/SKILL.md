## Workflow

1. **Scan and categorize** — Run `exec` to enumerate the target directory, grouping files by age (`mtime`), size, and folder type. Use `find` with `-type f -mtime +N` to flag stale artifacts. For scripts, also check `last` access time to determine dormancy.

2. **Apply preservation rules** — Honor explicit user preferences (e.g., "保留memory/ 入面啲backup"). For memory/ folders, always preserve anything inside `backup/` subdirectories. For scripts, check if the script is still referenced by TOOLS.md, task_router.js, smart_query.js, .registry/scripts.json, or any other active file before trashing.

3. **Identify all cross-references** — Before moving or renaming anything, grep the workspace for all files that reference the target script name:
   ```bash
   grep -rl "script_name" ~/.openclaw/workspace/ --include="*.js" --include="*.md"
   ```
   This catches TOOLS.md, .registry, task_router.js, smart_query.js, hermes_lessons_progress.js, SECURITY_AUDIT_REPORT.md, and any other docs that mention the script.

4. **Archive or trash in order** — Move stale scripts to `scripts/archive/` or `scripts/_legacy/` before trashing. Use `mv` to move to trash rather than `rm` for reversibility. Archive directories (not just files) by moving them into `_legacy/` subdirectories.

5. **Cascade-update all references** — For each file that references the renamed or trashed script:
   - If renamed: update the filename in usage text, header comments, and any `require()` or import statements
   - If trashed: remove the entry from TOOLS.md table, .registry/scripts.json, and any task router references
   - Verify zero residual mentions after each pass with a second `grep` scan

6. **Update registry and documentation** — Modify `.registry/scripts.json` to reflect the new filename or remove the entry. Update TOOLS.md usage examples and description columns. If TOOLS.md auto-generates (e.g., via cron at 00:41), note the auto-fix timing and let the cron handle it.

7. **Verify completeness** — Run a final `grep` for the old script name across all active scripts and docs. Report zero residual mentions as the completion signal.

## Pitfalls

- ⚠️ Updating one reference file but missing others — TOOLS.md updated but task_router.js still mentions the old name — the script appears broken to future sessions. Always grep comprehensively before declaring done.
- ⚠️ Trashing a script still actively referenced by a cron job — the cron fails silently next run. Check `openclaw cron list` for any job referencing the target script before trashing.
- ⚠️ Moving files to trash without preserving backups — `~/.Trash/` contents grow indefinitely. If the user said "keep memory/ backups," only touch `memory/` content after confirming it's truly backup (mtime old, no recent references).
- ⚠️ Assuming auto-generated docs will fix themselves immediately — SYMBOLS.md and similar files may be regenerated by cron at a fixed time. Don't report completion until that cron has run, or explicitly note the pending auto-fix.
- ⚠️ Editing .registry/scripts.json without also updating TOOLS.md — the registry and the user-facing docs go out of sync, causing "command not found" failures for the user.
- ⚠️ Archiving a script's only active version — e.g., archiving `stock_merge_pro.js` while `merge_show_stock_list.js` (the renamed version) hasn't been created yet. Always rename to the new name first, verify it works, then archive the old name.
